# Идея SIEM-Lite: позиция относительно enterprise SIEM

[Указатель `docs/`](README.md). Документ фиксирует **зачем** существует SIEM-Lite и **чем он отличается** от полноформатного вендорского SIEM. Детали архитектуры — в [`ARCHITECTURE.md`](ARCHITECTURE.md), таблица упрощений и roadmap — в [`RISKS_AND_ROADMAP.md`](RISKS_AND_ROADMAP.md).

---

## Зачем SIEM-Lite

**Enterprise SIEM** у крупных вендоров — это часто монолитная экосистема: десятки коннекторов из коробки, threat intel, UEBA, SOAR, отчёты под аудит, мультитенантность, формализованные SLA поддержки и лицензирование по объёму данных.

**SIEM-Lite** — целенаправленный **конвейер безопасности** под микросервисное приложение на **.NET 9 + React** и соседнюю инфраструктуру. Тот же **логический контур**, что у современного SIEM (сбор → парсинг → обогащение → очередь → хранение → детекция → корреляция → алерты → визуализация), но **меньше готовой «упаковки»** вокруг SOC-процессов, регуляторики и универсального маркетплейса интеграций.

Целевой масштаб и качество сервиса зафиксированы в корневом [`README.md`](../README.md): порядок **10k–50k EPS**, критические алерты **до ~30 с** от события, **парсинг p99 < 5 ms** на типовом событии (Rust-путь).

## Где SIEM-Lite совпадает с «настоящим» SIEM

| Слой | Роль |
|------|------|
| Инжест | Vector: агенты, агрегатор, маршрутизация по типам источников |
| Обработка | Парсинг, нормализация к ECS-подобной схеме, PII-маскирование, GeoIP/ASN и кэш идентичностей |
| Очередь | Kafka-совместимый Redpanda, идемпотентные продюсеры, consumer groups |
| Хранение | ClickHouse, партиции, TTL, холод на S3-совместимом хранилище, дедуп по `event_id` |
| Детекция | Движок правил + stateful-коррелятор (окна, Redis) |
| Алертинг | Alertmanager, маршрутизация и группировка |
| Наблюдаемость платформы | Prometheus, Loki, OpenTelemetry |

Это уже не «просто централизованные логи», а **связка уровня SIEM по компонентам**.

## Масштабные отличия от enterprise

| Область | SIEM-Lite (фокус сейчас) | Типичный enterprise SIEM |
|---------|-------------------------|---------------------------|
| Корреляция | Относительно простые временные окна и сценарии | Многошаговые цепочки, граф сущностей, тяжёлая UEBA |
| Threat intelligence | **intel-connector** + Redis/парсер; без VirusTotal/OTX «из коробки» (roadmap) | Постоянное обогащение по IoC из множества внешних API |
| SOAR | Webhook, инцидент-трекинг снаружи; без playbooks «из коробки» | Оркестрация действий и стандартные runbook-и |
| ML / поведение | Пороги и простые аномалии; UEBA в Phase 3 | Зрелая entity analytics |
| Мультитенантность | Один контур данных | Изоляция tenant на уровне продукта |
| Compliance | Нет готовых пакетов отчётов под стандарты | Готовые шаблоны (SOC2, PCI, HIPAA и т.д.) |
| Парсеры | Ограниченный набор источников + ручное развитие (VRL и код) | Сотни pre-built парсеров |
| Управление политиками | Конфигурация в SQL/YAML | Веб-интерфейсы для retention, ролей, исключений |
| Высокая доступность | Поэтапное усложнение (в т.ч. single-node CH на старте) | Кластеры и формальные RTO/RPO как норма |
| Семантика доставки | At-least-once на участках, дедуп при записи; exactly-once на всём пути не цель | Иногда жёстче требования к доказуемости целостности цепочки |

Итог в одну фразу: **отличает не отсутствие SIEM как класса системы, а глубина готовых SOC- и enterprise-фич и универсальность под любой ландшафт.**

План масштабирования **10k → 50k EPS** опирается на горизонтальное масштабирование stateless-компонентов, рост партиций Redpanda и **шардирование ClickHouse** с распределёнными таблицами. Конкретные узкие места и шаги — в разделе про масштабирование в [`RISKS_AND_ROADMAP.md`](RISKS_AND_ROADMAP.md).

## Эволюция: Lite → Standard → Enterprise (кратко)

- **Phase 1** — рабочий контур: сбор, парсинг, хранение, детекция, алерты, Grafana.
- **Phase 2** — live threat intel (MISP и т.д.), HA ClickHouse, LDAP/RBAC, усложнение коррелятора, при необходимости интеграция с внешними IR-платформами (например TheHive) поверх текущего case-management-rs.
- **Phase 3** — UEBA (ML), SOAR (n8n/Shuffle), расширенные отчёты под compliance, больше парсеров (облака, Windows, CEF/LEEF).

Полный список — в roadmap того же [`RISKS_AND_ROADMAP.md`](RISKS_AND_ROADMAP.md).
